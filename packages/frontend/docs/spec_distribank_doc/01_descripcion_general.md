# Descripción General del Sistema — DistriBank

DistriBank es un sistema bancario de base de datos diseñado para modelar las operaciones financieras características de empresas de *retail financiero* como Coppel o Liverpool, cubriendo tanto el manejo de efectivo mediante cuentas de débito como el otorgamiento y gestión de crédito. El esquema adopta una arquitectura de base de datos única (*single-node*) como punto de partida deliberado —antes de introducir distribución, el sistema debe funcionar de forma correcta y observable en su forma más simple— estableciendo así una línea base de comportamiento contra la cual comparar el impacto de las decisiones de distribución posteriores.

El modelo de datos se estructura en seis tablas con responsabilidades claramente delimitadas: `customers`, `customer_accounts`, `accounts`, `cards`, `transactions` y `transaction_log`. La relación entre estas entidades refleja el flujo operativo de una institución financiera real, donde un cliente puede poseer hasta una cuenta de débito (*checking*) y hasta una cuenta de crédito, cada una con sus propias tarjetas, y toda actividad financiera queda registrada tanto en la tabla de transacciones como en un log de eventos auditables.

## Seguridad

La seguridad del esquema opera en dos niveles distintos pero complementarios: integridad de los datos financieros mediante restricciones de base de datos, y control estructural sobre las relaciones entre entidades.

El nivel más crítico es la integridad financiera. La tabla `accounts` establece una separación semántica estricta entre cuentas *checking* y cuentas de crédito mediante un conjunto de `CHECK` constraints que hacen que la base de datos rechace, de forma activa, cualquier estado de cuenta incoherente. Una cuenta de crédito debe tener siempre valores no nulos en `credit_limit` y `available_credit`, mientras que `overdraft_limit` debe ser necesariamente nulo para ese tipo; la lógica inversa aplica para cuentas *checking*. Esto impide que reglas de negocio críticas —como el límite de crédito autorizado— dependan únicamente de validaciones en capa de aplicación, que son intrínsecamente más frágiles ante bugs, *deployments* defectuosos o acceso directo a la base de datos. Complementariamente, el constraint `CHECK (balance >= 0 OR account_type = 'CREDIT')` formaliza que solo las cuentas de crédito pueden operar con saldo negativo, comportamiento esperado cuando el titular ha dispuesto crédito.

En la tabla `transactions` se implementan tres constraints de seguridad operacional. El primero garantiza que toda transacción involucre un monto estrictamente positivo (`amount > 0`), bloqueando inserciones que podrían usarse para mover valor de forma no convencional. El segundo previene que una cuenta sea simultáneamente origen y destino de la misma transacción (`from_account_id != to_account_id`), evitando una clase de inconsistencias donde el sistema procesaría un movimiento sin transferencia real de valor. El tercero asegura que si una transacción se realiza con tarjeta, esa tarjeta pertenezca a la cuenta de origen —un control de autorización fundamental que previene el uso indebido de instrumentos de pago de terceros.

La tabla `cards` extiende esta lógica con un constraint que hace cumplir la correspondencia entre el tipo de tarjeta y el tipo de cuenta asociada: una tarjeta de débito únicamente puede vincularse a una cuenta *checking*, y una de crédito solo a una cuenta de crédito. Este control cierra un vector de inconsistencia donde un instrumento de pago podría quedar asociado a un producto financiero incompatible. Adicionalmente, el campo `cvv` se almacena directamente en la tabla como parte del modelo —detalle que en producción real requeriría *hashing* o tokenización, y que en este contexto educativo es un punto de atención deliberado para discutir estrategias de seguridad de datos sensibles.

La tabla `transaction_log` actúa como capa de no repudio. Al registrar cada evento del ciclo de vida de una transacción (`INITIATED`, `DEBIT_APPLIED`, `CREDIT_APPLIED`, `COMPLETED`, `COMPENSATED`, `FAILED`), el sistema mantiene un *audit trail* que permite reconstruir exactamente qué ocurrió durante una operación fallida o disputada. El campo `details` de tipo `jsonb` provee flexibilidad para adjuntar metadatos específicos de cada evento sin romper el esquema, lo cual es especialmente valioso cuando se investigan incidentes en producción.

## Disponibilidad

La disponibilidad del sistema, en su fase centralizada, se aborda principalmente mediante una estrategia de indexación que reduce la latencia de lectura en los *paths* de acceso más críticos, y mediante decisiones de modelado que minimizan la contención transaccional.

Cada tabla cuenta con índices definidos sobre los campos de acceso más frecuentes en el contexto bancario. En `accounts`, los índices sobre `account_number` y `status` cubren los dos patrones de lectura más comunes: la búsqueda de una cuenta por su número (operación de punto de venta y cajero automático) y el filtrado por estado para operaciones de reconciliación o bloqueo masivo. Los índices sobre `week_transactions` y `last_limit_increase_at` soportan procesos *batch* periódicos, como revisión automática de límites de crédito o detección de cuentas con actividad inusual.

En `cards`, el índice compuesto sobre `(expiration_date, status)` está diseñado para soportar eficientemente los procesos de renovación automática y notificación de tarjetas próximas a vencer, que generalmente se ejecutan como queries del tipo `WHERE expiration_date <= :fecha AND status = 'ACTIVE'`. En `transactions`, el índice compuesto `(status, initiated_at)` permite consultas eficientes sobre transacciones ordenadas cronológicamente, patrón central en procesos de liquidación y monitoreo de transacciones colgadas.

El campo `transaction_uuid` en la tabla `transactions` merece atención especial desde la perspectiva de disponibilidad. Al generar un UUID en el momento de iniciación (`gen_random_uuid()`) antes de que la transacción se complete, el sistema dispone de un identificador idempotente que puede usarse para detectar reintentos duplicados. En una arquitectura donde el cliente puede reintentar una solicitud por *timeout* de red, la capacidad de identificar si esa transacción ya fue procesada —mediante búsqueda por UUID— es fundamental para no cobrar dos veces una operación.

La tabla `customer_accounts` como tabla puente entre `customers` y `accounts` también contribuye a la disponibilidad al simplificar la estructura de consultas de perfil de cliente. En lugar de un modelo donde `accounts` tendría una *foreign key* directa a `customers` —y las consultas de perfil requirieran filtrar por `customer_id` sobre una tabla potencialmente muy grande—, esta indirección permite obtener los identificadores de cuenta de un cliente con un *lookup* puntual sobre una tabla de cardinalidad controlada (máximo un registro por cliente).

En la fase distribuida, la disponibilidad adquiere una dimensión adicional. El esquema VIP alojado en Nodo C (`distribank_vip_customers`) funciona como réplica de alta disponibilidad para los clientes clasificados como *VIP* —aquellos cuya suma de `week_transactions` sobre cuentas activas alcanza o supera 3— permitiendo redirigir operaciones críticas cuando el nodo primario de un cliente no responde. Esta arquitectura introduce deliberadamente el problema de *replication lag* y la ventana de inconsistencia entre ciclos de sincronización como tensiones académicamente observables.

## Flexibilidad

La flexibilidad del esquema se manifiesta en tres dimensiones: adaptabilidad del modelo de productos financieros, extensibilidad del sistema de eventos, y preparación para escenarios de distribución futura.

El diseño de la tabla `accounts` utiliza un patrón de atributos opcionales condicionados por tipo (`account_type`) en lugar de tablas separadas por producto. Esto permite que una sola entidad represente dos productos financieros con comportamientos distintos —crédito y débito— sin duplicar la lógica de transacciones ni los índices operacionales. Los constraints `CHECK` que gobiernan qué campos son nulos según el tipo actúan como documentación ejecutable del modelo de negocio: explicitan en el esquema mismo las reglas que diferencian ambos productos. Esta decisión facilita agregar un tercer tipo de cuenta en el futuro (por ejemplo, una cuenta de ahorro con su propio conjunto de atributos) extendiendo el constraint existente y añadiendo columnas opcionales, sin reestructurar las tablas que ya dependen de `accounts`.

La tabla `cards` adopta el mismo principio para soportar múltiples tarjetas por cuenta —principal y adicionales— sin ninguna distinción estructural entre ellas. La relación entre `cards` y `accounts` es de muchos-a-uno. La lógica de negocio sobre cuántas tarjetas son válidas simultáneamente o cuáles están activas se resuelve mediante `status`, lo que confiere flexibilidad para implementar productos como tarjetas adicionales para familiares o tarjetas virtuales de un solo uso sin modificar el esquema.

El campo `details jsonb` en `transaction_log` es quizás el punto de mayor flexibilidad explícita del esquema. El tipo `jsonb` de PostgreSQL permite almacenar documentos semiestructurados consultables sin definir columnas adicionales, lo que significa que distintos tipos de eventos pueden enriquecer su registro con metadatos heterogéneos. Un evento `DEBIT_APPLIED` podría registrar el saldo previo, un evento `COMPENSATED` podría incluir el identificador de la transacción original, y un evento `FAILED` podría contener el código de error del sistema externo involucrado —todo sin alterar el DDL.

Finalmente, la presencia del `transaction_log` con sus tipos de evento (`INITIATED`, `DEBIT_APPLIED`, `CREDIT_APPLIED`, `COMPLETED`, `COMPENSATED`, `FAILED`) revela la preparación del esquema para soportar patrones de transacción distribuida. Los eventos `COMPENSATED` y el estado `ROLLED_BACK` en `transactions` anticipan la implementación del patrón SAGA, donde una transacción fallida en un nodo requiere ejecutar transacciones compensatorias en nodos anteriores para restituir la consistencia. Esta estructura de log es también compatible con el patrón de *Two-Phase Commit* (2PC), donde el log actúa como registro de preparación y confirmación de cada participante. El hecho de que ambos patrones compartan la misma infraestructura de *logging* es una decisión de diseño que preserva opcionalidad: el sistema puede evolucionar hacia cualquiera de los dos mecanismos —o hacia una combinación de ambos— sin rediseñar el modelo de eventos.
